Los Principios de Diseño SOLID

SOLID es un acronimo que soporta los siguientes principios de diseño (y sus abreviatuas)

  • Principio de Resposabilidad Unica. Single Responsibility Principle (SRP)
  • Principio de Abierto-Cerrado. Open-Closed Principle (CSP)
  • Principio de Sustitución de Liskov. Liskov Substitution Principle (LSP)
  • Principio de Segregacion de Interfaz. Interface Segregation Principle (ISP)
  • Principio de Inversion de Dependencia. Dependency Inversion Principle (DIP)

Estos principios son introducidos por Robert C. Martin a principio de los 2000s - En efecto, aquellas son una seleccion de cinco principios de docenas que son expresados en los libros y blogs de Robert. Estos cincos particulares temas introducen la discusión en general del diseño de patrones y software, por lo tanto, antes de lanzarnos en Patrones de Diseño, vamos a realizar un breve resumen acerca de lo que son los Principios SOLID

Principio de Unica Responsabilidad

Supongamos que decides tener un diario de vuestras ideas/opiniones mas intimas. El diario es usado para guardar numeros de entrada. Podrias modelarlo de la siguiente forma: 


    public class Journal
    {
        private readonly List<string> entries = new();
        
        // Solo un contador para el total de numeros de entradas
        private static int count = 0;
    }

Ahora, podrias agregar funcionalidad para añadir una entrada a la revista, procedido del numero de entrada de la revista. Tambien podrias tener funcionalidad para remover entradas (Implementado de una forma muy ordinaria de la siguiente forma). Esto es facil: 

    
    public void AddEntry(string text)
    {
        entries.Add($"{++count}: {text}");
    }

    public void RemoveEntry(int index)
    {
        entries.RemoveAt(index);
    }

y ahora usamos el diario de la siguiente forma: 

        
    var j = new Journal();
    j.AddEntry("I cried today.");
    j.AddEntry("I ate a bug.")

Tiene sentido que este metodo forme parte de la clase Journal porque añadir una entrada al diario es algo que realmente el diario tiene que hacer. Es responsabilidad del diario mantener las entradas, por lo que cualquier cosa relacionada con eso es juego limpio.

Ahora, suponga que decide hacer que el diario percista guardando en un archivo. Agrege este codigo a la clase Journal: 

    
    public void Save(string filename, bool overwrite = false)
    {
        File.WriteAllText(filename, ToString());
    }

Este enfoque es problematico. La responsabilidad de los Diarios (de la clase diario) es guardar las entradas, no escribirlas en el disco. Si agregas la funcionalidad de persistencia a la clase Journal y a clases similares, cualquier cambio en el enfoque hacia su persistencia (digamos, usted decide si escribir en la nube en vez del disco) requeriría muchos pequeños cambios en cada una de las clases afectadas.
